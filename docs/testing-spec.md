# Testing Specification

Актуально для состояния проекта на 10 апреля 2026 года.

## Назначение Документа

Этот документ описывает функциональную спецификацию проекта с точки зрения тестирования.

Документ нужен, чтобы тестировщик мог:

- понять состав системы;
- подготовить окружение;
- проверить основные пользовательские сценарии;
- проверить backend API;
- понимать ожидаемое поведение и ограничения системы.

## Объект Тестирования

В состав системы входят:

- Android-приложение каталога товаров;
- backend/BFF на `Node.js + TypeScript`;
- `PostgreSQL` для хранения source-данных, переводов, отзывов и валютных курсов;
- внешние источники `DummyJSON` и `Frankfurter`.

## Состав Репозитория

- `app` — Android host-модуль;
- `feature` — shared UI и бизнес-логика каталога;
- `model` — domain-модели и контракты;
- `backend` — backend/BFF;
- `docs` — документация и скриншоты.

## Поддерживаемая Платформа

### Клиент

- Android;
- минимальная версия Android: `API 26` / Android 8.0;
- основной сценарий локального запуска: Android Emulator.

### Backend

- Node.js `20+`;
- PostgreSQL `16+` или совместимая версия, используемая через Docker Compose.

## Что Входит В Scope Тестирования

- запуск backend и его доступность;
- загрузка каталога в мобильном приложении;
- поиск;
- фильтрация по категориям;
- сортировка;
- локализация интерфейса и каталога;
- выбор валюты;
- серверная пагинация;
- открытие карточки товара;
- отображение отзывов;
- error/empty/loading states;
- основные backend endpoints.

## Что Не Входит В Scope

- авторизация и профили пользователей;
- корзина, checkout, оплата;
- избранное;
- push-уведомления;
- офлайн-режим;
- админский интерфейс;
- редактирование каталога из UI;
- iOS-клиент.

## Внешние Зависимости И Ограничения

- source-каталог и source-отзывы берутся из `DummyJSON`;
- валютные курсы берутся из `Frankfurter`;
- точное количество товаров и категорий может меняться вместе с внешним датасетом;
- source-язык данных всегда `en`;
- source-валюта данных всегда `USD`;
- переводы каталога и отзывов являются demo-локализацией и заполняются статически на backend;
- переводы не являются production-grade machine translation.

## Окружение Для Тестирования

### Рекомендуемое Окружение

- Windows/macOS/Linux workstation;
- Docker Desktop или совместимый Docker Engine;
- Android Studio;
- Android Emulator;
- JDK `17+`.

### Минимальные Предусловия

Для проверки приложения должны быть доступны:

- `backend` на `http://localhost:8080`;
- `PostgreSQL`;
- сеть до `DummyJSON` и `Frankfurter`.

### Важное Поведение На Старте Backend

При каждом старте backend:

1. проверяет/создает схему БД;
2. полностью перезагружает source-каталог из `DummyJSON`;
3. пересеивает переводы;
4. проверяет валютные курсы.

Из этого следуют два правила для тестирования:

- первый старт backend может быть дольше обычного;
- если внешний источник недоступен, backend может не подняться корректно.

## Функциональная Спецификация

## 1. Запуск Приложения И Начальная Загрузка

### Описание

После запуска приложение открывает экран каталога и запрашивает первую страницу данных с backend.

### Ожидаемое Поведение

- при первом открытии показывается loading state;
- после успешной загрузки отображается список товаров;
- стартовые значения:
  - язык: `en`, если ранее не был сохранен другой язык;
  - валюта: `USD`;
  - категория: `All`;
  - сортировка: `Price: low to high`;
  - номер страницы: `1`;
  - размер страницы: `20`.

### Что Проверять

- приложение не падает на старте;
- loading state сменяется контентом;
- список товаров отображается;
- UI-контролы доступны для взаимодействия.

## 2. Поиск

### Описание

Пользователь вводит текст в строку поиска.

### Ожидаемое Поведение

- запрос отправляется не мгновенно, а с `debounce 300ms`;
- поиск выполняется по локализованным `title` и `description`;
- изменение поиска сбрасывает список к первой странице;
- пустая строка поиска возвращает обычный каталог без текстового фильтра.

### Что Проверять

- при быстром вводе приложение не дергает UI хаотично;
- после паузы в `300ms` результат обновляется;
- результаты соответствуют введенному запросу;
- очистка поля возвращает полный список.

## 3. Фильтрация По Категории

### Описание

Пользователь выбирает категорию из chips-блока.

### Ожидаемое Поведение

- в списке категорий всегда есть `All`;
- при выборе конкретной категории каталог фильтруется на сервере;
- при выборе `All` фильтр по категории снимается;
- изменение категории сбрасывает список к первой странице.

### Что Проверять

- выбранная категория визуально выделяется;
- после смены категории список обновляется;
- товары соответствуют выбранной категории;
- возврат к `All` убирает категорийный фильтр.

## 4. Сортировка

### Доступные Варианты

- `Price: low to high`
- `Price: high to low`
- `Rating: high to low`

### Ожидаемое Поведение

- сортировка применяется на сервере;
- смена сортировки сбрасывает список к первой странице;
- порядок элементов соответствует выбранному режиму.

### Что Проверять

- dropdown открывается и закрывается корректно;
- выбранное значение отображается в контроле;
- после смены сортировки данные реально меняют порядок.

## 5. Выбор Языка

### Описание

Пользователь меняет язык через dropdown `Language`.

### Ожидаемое Поведение

- меняются UI-строки интерфейса;
- backend запрашивается с новым `lang`;
- названия товаров, описания, категории и отзывы приходят локализованными;
- выбранный язык сохраняется и применяется после перезапуска приложения.

### Что Проверять

- меняется подпись UI-контролов;
- обновляется контент каталога;
- после restart приложения выбранный язык остается тем же.

## 6. Выбор Валюты

### Описание

Пользователь меняет валюту через dropdown `Currency`.

### Ожидаемое Поведение

- backend запрашивается с новым `currency`;
- цены в каталоге и карточке товара обновляются;
- `USD` является source-валютой;
- для остальных валют backend использует серверную конвертацию;
- выбор валюты не сбрасывает язык.

### Что Проверять

- меняется валютный символ и значение цены;
- одна и та же карточка товара показывает другую цену при смене валюты;
- переход на detail-экран использует текущую валюту.

### Ограничение

Валюта в текущей реализации не обязана сохраняться между cold start приложения.

## 7. Серверная Пагинация

### Описание

Каталог подгружается страницами с backend.

### Ожидаемое Поведение

- первая страница загружается автоматически;
- следующая страница подгружается при приближении к концу списка;
- во время дозагрузки показывается нижний loader;
- уже загруженные элементы не должны дублироваться;
- когда страниц больше нет, подгрузка прекращается.

### Что Проверять

- размер страницы соответствует `20`;
- после скролла вниз список увеличивается;
- loader отображается только во время дозагрузки;
- при достижении конца датасета новые страницы больше не запрашиваются.

## 8. Карточка Товара

### Описание

Пользователь нажимает на любой товар в каталоге и открывает detail-экран.

### Ожидаемое Поведение

- переход происходит по тапу на карточку товара;
- detail-экран содержит:
  - кнопку возврата назад;
  - изображение;
  - категорию;
  - рейтинг;
  - цену;
  - название;
  - описание;
  - блок отзывов или empty state для отзывов;
- detail-экран использует текущие язык и валюту из каталога.

### Что Проверять

- переход открывает правильный товар;
- кнопка `Back to catalog` работает;
- после возврата сохраняются текущее состояние каталога и выбранные фильтры;
- карточка не показывает технические route-ошибки пользователю.

## 9. Отзывы

### Описание

Отзывы приходят с backend как часть карточки товара.

### Ожидаемое Поведение

- отзывы отображаются отдельными карточками;
- показываются имя автора, дата, рейтинг и комментарий;
- количество отзывов в заголовке не отображается;
- UI показывает не более `3` review cards;
- если отзывов нет, показывается карточка `No reviews yet`.

### Что Проверять

- отзывы присутствуют у товаров, где они есть в dataset;
- язык отзывов соответствует текущему `lang`;
- отображается не больше трех отзывов;
- блок отзывов выглядит корректно для loaded и empty сценариев.

## 10. Error State И Retry На Экране Каталога

### Описание

Если каталог не загрузился и список пустой, приложение показывает error state.

### Ожидаемое Поведение

- отображается error card;
- доступна кнопка `Retry`;
- при восстановлении backend повторная попытка должна загрузить каталог.

### Что Проверять

- отключить backend до запуска приложения;
- убедиться, что отображается ошибка;
- вернуть backend;
- нажать `Retry`;
- убедиться, что каталог загрузился.

## 11. Error State И Retry В Карточке Товара

### Описание

Если detail-запрос завершился ошибкой, отображается отдельное состояние ошибки.

### Ожидаемое Поведение

- на detail-экране доступна кнопка назад;
- показывается карточка с сообщением `Could not load this product`;
- доступна кнопка `Retry`;
- после восстановления backend повтор должен загрузить карточку товара.

### Что Проверять

- открыть каталог;
- после этого сделать backend недоступным;
- нажать на товар;
- убедиться, что показывается error state detail-экрана;
- вернуть backend и нажать `Retry`.

## 12. Empty State

### Описание

Если комбинация поиска и фильтров не вернула результатов, показывается empty state.

### Ожидаемое Поведение

- отображается `Nothing found`;
- доступна кнопка `Reset filters`;
- `Reset filters` очищает:
  - search query;
  - category filter;
  - sort option;
- `Reset filters` не меняет:
  - language;
  - currency.

### Что Проверять

- подобрать запрос, который не дает результатов;
- убедиться, что появляется empty state;
- нажать `Reset filters`;
- проверить, что каталог снова отображается;
- проверить, что язык и валюта сохранились.

## 13. Поведение UI При Отсутствии Фокуса

### Описание

При тапе вне строки поиска фокус должен сниматься.

### Что Проверять

- активировать поле поиска;
- нажать в пустую часть экрана;
- убедиться, что фокус снят и экран остается рабочим.

## Backend API Specification Для Тестирования

## 1. `GET /health`

### Ожидаемое Поведение

- HTTP `200`;
- тело ответа:

```json
{
  "status": "ok"
}
```

## 2. `GET /api/v1/languages`

### Ожидаемое Поведение

- HTTP `200`;
- возвращается список из поддерживаемых языков;
- `en` помечен как `isSourceLanguage = true`.

## 3. `GET /api/v1/currencies`

### Ожидаемое Поведение

- HTTP `200`;
- возвращается список поддерживаемых валют;
- `USD` помечен как `isSourceCurrency = true`.

## 4. `GET /api/v1/catalog`

### Параметры

- `lang`
- `currency`
- `page`
- `pageSize`
- `query`
- `category`
- `sort`

### Ожидаемое Поведение

- HTTP `200` при валидном запросе;
- `pageSize` не должен превышать `100`;
- возвращается структура `language + currency + categories + items + meta`;
- `meta.currentPage`, `meta.pageSize`, `meta.totalPages` должны быть согласованы с ответом;
- при `currency != USD` поле `meta.exchangeRateProvider` должно быть `frankfurter`;
- при `currency == USD` поле `meta.exchangeRateProvider` должно быть `null`.

### Ошибочные Сценарии

- неверный `lang` -> `400`;
- неверный `currency` -> `400`;
- `page < 1` -> `400`;
- `pageSize < 1` или `pageSize > 100` -> `400`.

## 5. `GET /api/v1/catalog/:productId`

### Параметры

- `productId` в path;
- `lang`;
- `currency`.

### Ожидаемое Поведение

- HTTP `200` для существующего товара;
- возвращаются `product`, `reviews`, `meta`;
- цена товара соответствует выбранной валюте;
- отзывы локализуются по выбранному языку.

### Ошибочные Сценарии

- несуществующий `productId` -> `404`;
- неверный `lang` или `currency` -> `400`.

## Общий Формат Ошибок Backend

Backend должен возвращать ошибки в формате:

```json
{
  "statusCode": 400,
  "error": "BAD_REQUEST",
  "message": "..."
}
```

## Рекомендованный Smoke Checklist

1. Поднять backend и убедиться, что `/health` отвечает `200`.
2. Открыть приложение и дождаться загрузки каталога.
3. Проверить поиск по произвольному валидному запросу.
4. Проверить фильтр по категории.
5. Проверить все три варианта сортировки.
6. Сменить язык и убедиться, что обновились UI и данные.
7. Сменить валюту и убедиться, что изменилась цена.
8. Пролистать список вниз и проверить подгрузку следующей страницы.
9. Открыть карточку товара и проверить возврат назад.
10. Проверить наличие отзывов в detail-экране.
11. Отключить backend и проверить error state каталога.
12. Проверить `Retry` после восстановления backend.
13. Проверить detail error state при недоступности backend.
14. Проверить `/api/v1/languages`, `/api/v1/currencies`, `/api/v1/catalog`, `/api/v1/catalog/:productId` через Swagger.

## Acceptance Criteria

Система считается соответствующей спецификации, если:

- приложение стабильно запускается и не падает в основных сценариях;
- каталог загружается с backend;
- работают поиск, фильтрация, сортировка, локализация и выбор валюты;
- detail-экран товара открывается по тапу на карточку;
- detail-экран содержит отзывы или корректный empty state;
- error и retry сценарии отрабатывают и в каталоге, и в карточке товара;
- backend endpoints отвечают в ожидаемом формате;
- backend-документация и корневая документация соответствуют реальному поведению системы.

## Known Limitations

- проект является demo-приложением, а не production e-commerce системой;
- переводы каталога и отзывов статические и могут звучать неидеально;
- source-данные зависят от внешнего `DummyJSON`;
- приложение требует доступный backend;
- UI detail-экрана показывает максимум `3` отзыва;
- exact product count не является жестко зафиксированным контрактом, потому что зависит от внешнего датасета.
