# Спецификация фичи экрана карточки товара

## Зачем нужен этот документ

Документ описывает добавление фичи detail-экрана товара в существующее приложение каталога.
Фича включает:

- новую backend-ручку для получения данных по конкретному товару;
- изменения в shared KMP-слое (`model` и `feature`);
- новый пользовательский сценарий открытия карточки товара из каталога.

Документ нужен как единая точка для постановки задачи, согласования API-контракта и проверки ожидаемого поведения мобильного клиента.

## Контекст

В приложении уже есть экран каталога товаров с поиском, фильтрацией, сортировкой, пагинацией, выбором языка и валюты.
Следующий шаг развития фичи — дать пользователю возможность открыть отдельный экран товара и получить расширенную информацию по выбранной карточке.

## Цель фичи

После нажатия на товар в каталоге пользователь должен перейти на отдельный экран карточки товара и увидеть:

- изображение товара;
- название;
- категорию;
- рейтинг;
- цену в выбранной валюте;
- описание;
- отзывы по товару;
- кнопку возврата назад в каталог.

## Что входит в scope

- backend endpoint для запроса данных одного товара;
- получение локализованных данных товара и отзывов;
- конвертация цены в выбранную валюту;
- новый detail-экран в KMP UI;
- loading/error/empty-состояния detail-экрана;
- возврат назад без потери состояния каталога.

## Что не входит в scope

- корзина, избранное, checkout;
- редактирование товара;
- related products / recommendations;
- offline cache detail-экрана;
- обязательное внедрение отдельного navigation framework, если текущей экранной модели достаточно.

## Пользовательский сценарий

1. Пользователь открывает каталог товаров.
2. Пользователь нажимает на карточку товара.
3. Приложение открывает detail-экран и показывает состояние загрузки.
4. Клиент запрашивает backend по `productId` с текущими `lang` и `currency`.
5. Backend возвращает локализованную карточку товара и список отзывов.
6. Приложение отображает данные на отдельном экране.
7. Пользователь нажимает кнопку возврата назад.
8. Приложение возвращает пользователя в каталог без сброса текущих фильтров, строки поиска и уже загруженного списка.

## Backend-часть

### Новая ручка

`GET /api/v1/catalog/:productId`

### Назначение ручки

Ручка возвращает detail-данные одного товара для экрана карточки товара.

### Path-параметры

- `productId` — идентификатор товара из каталога.

### Query-параметры

- `lang` — код языка ответа, по умолчанию `en`;
- `currency` — код валюты ответа, по умолчанию `USD`.

### Успешный ответ

```json
{
  "language": "ru",
  "currency": "EUR",
  "product": {
    "id": "10",
    "title": "Ночная сыворотка",
    "description": "Легкая сыворотка для вечернего ухода.",
    "price": {
      "amount": 18.19,
      "currency": "EUR"
    },
    "rating": 4.8,
    "imageUrl": "https://example.com/serum.webp",
    "category": {
      "slug": "beauty",
      "title": "Красота"
    }
  },
  "reviews": [
    {
      "id": "review-1",
      "rating": 5,
      "comment": "Очень рекомендую!",
      "date": "2026-04-10T10:15:00.000Z",
      "reviewerName": "Emma Wilson"
    }
  ],
  "meta": {
    "sourceLanguage": "en",
    "sourceCurrency": "USD",
    "exchangeRateProvider": "frankfurter"
  }
}
```

### Ошибочные ответы

- `400 BAD_REQUEST` — невалидный `lang`, `currency` или `productId`;
- `404 PRODUCT_NOT_FOUND` — товар не найден;
- `503 SOURCE_CATALOG_UNAVAILABLE` — source-каталог еще не готов;
- `503 EXCHANGE_RATES_UNAVAILABLE` — курсы валют еще не готовы;
- `502` — ошибка внешнего провайдера;
- `500` — внутренняя ошибка backend.

### Требования к backend-логике

- источник товара — данные, синхронизированные в PostgreSQL;
- если выбран язык, отличный от source-языка, backend возвращает локализованные поля товара и локализованные отзывы;
- если выбрана валюта, отличная от source-валюты, backend конвертирует цену перед ответом;
- отзывам не требуется отдельная пагинация в рамках первой версии фичи;
- ручка должна использовать тот же формат ошибок, что и остальные endpoints каталога.

## KMP / Mobile-часть

### Изменения в `model`

Нужно ввести или использовать доменные сущности для detail-сценария:

- `ProductDetails`;
- `ProductReview`;
- существующую модель `Product` для общей карточки товара.

Также `ProductRepository` должен поддерживать метод получения detail-данных по `productId`, `languageCode` и `currencyCode`.

### Изменения в `feature`

Нужно добавить:

- state для detail-экрана;
- UI detail-экрана;
- обработку открытия карточки товара из списка;
- обработку retry для detail-запроса;
- обработку возврата назад в каталог.

### Требования к экрану

Экран должен показывать:

- кнопку возврата назад;
- hero-блок с изображением товара;
- название товара;
- категорию;
- рейтинг;
- цену;
- описание;
- список отзывов;
- empty state, если отзывов нет.

### Требования к состояниям

Должны поддерживаться состояния:

- `Hidden` — detail-экран не открыт;
- `Loading` — идет загрузка карточки товара;
- `Content` — товар успешно загружен;
- `Error` — detail-запрос завершился ошибкой.

### Правила поведения

- detail-запрос выполняется только после явного выбора товара;
- detail-экран использует текущие выбранные язык и валюту каталога;
- при возврате назад пользователь видит каталог в том же состоянии, в котором он был до открытия карточки;
- если detail-запрос завершился ошибкой, пользователь видит отдельный error state и кнопку `Retry`;
- если у товара нет отзывов, пользователь видит корректный empty state вместо пустого списка.

## UX-ограничения первой версии

- показывать не более трех отзывов на detail-экране;
- не добавлять отдельные tabs или секции со сложной навигацией;
- не добавлять редактирование, share и secondary actions;
- не требовать отдельного preload detail-данных из списка.

## Архитектурные требования

- `app` остается Android host-слоем;
- shared-логика detail-сценария располагается в `feature`;
- доменные контракты остаются в `model`;
- backend остается единственной точкой входа для клиента;
- detail-экран не должен ходить напрямую во внешние API.

Цепочка зависимостей не меняется:

```text
app -> feature -> model
```

Поток данных для detail-сценария:

```text
Product card tap -> KMP feature -> backend -> PostgreSQL
                                       -> currency rates
```

## Acceptance criteria

- при нажатии на товар в каталоге открывается отдельный detail-экран;
- detail-экран показывает loading state до прихода ответа;
- detail-экран отображает название, цену, рейтинг, категорию, описание и изображение товара;
- detail-экран использует текущие выбранные язык и валюту;
- отзывы отображаются на том же языке, что и карточка товара;
- при отсутствии отзывов отображается empty state;
- при ошибке detail-запроса отображается error state с кнопкой `Retry`;
- кнопка возврата возвращает в каталог без сброса фильтров и списка;
- backend ручка возвращает корректный JSON-контракт и ожидаемые коды ошибок.

## Требования к тестам

### Backend

- unit-тесты на route;
- unit-тесты на service;
- integration-тесты на repository / SQL-выборку detail-данных;
- тест на `404`, если товар не найден.

### KMP / Feature

- unit-тесты на открытие detail-экрана;
- unit-тесты на успешную загрузку товара;
- unit-тесты на error state и retry;
- unit-тесты на сохранение состояния каталога после возврата назад.

### Ручная проверка

- открыть товар из каталога;
- проверить loading state;
- проверить возврат назад;
- проверить смену языка и валюты перед открытием detail-экрана;
- проверить товар с отзывами;
- проверить товар без отзывов;
- проверить error state при недоступном backend.
