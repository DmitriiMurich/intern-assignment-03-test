# Архитектурные решения

## Зачем этот файл

Здесь фиксируются инженерные решения по проекту, чтобы потом было проще объяснить архитектуру и спорные места на собеседовании.

## Подтверждённые решения

### D-001. Проект переведён из Android-only шаблона в KMP multi-module структуру

- Причина: это прямое требование задания.
- Следствие: `app` играет роль Android host-модуля, а shared-логика живёт в `model` и `feature`.

### D-002. `app -> feature -> model` — основная цепочка зависимостей

- Причина: Android-слой не должен быть местом хранения бизнес-логики.
- Следствие: `app` только поднимает DI и запускает экран, а сама логика каталога изолирована в shared-модулях.

### D-003. Состояние экрана хранится в `StateFlow`

- Причина: экрану нужен один наблюдаемый источник правды.
- Следствие: Compose подписывается на состояние как на single source of truth.

### D-004. Одноразовые UI-события при необходимости должны идти через `SharedFlow`

- Причина: transient-события не должны жить в постоянном UI state.
- Следствие: на собеседовании можно чётко развести роли `StateFlow` и `SharedFlow`.

### D-005. Поиск, фильтрация, сортировка и пагинация реализуются вне UI

- Причина: это основная бизнес-логика задания.
- Следствие: её можно тестировать без Android framework.

### D-006. Приложение использует реальную сеть, тесты будут изолироваться через mock или fake репозиторий

- Причина: задание явно подразумевает Ktor Client и одновременно требует unit-тесты с контролируемым поведением.
- Следствие: runtime и тестовый контур разделены по ответственности.

### D-007. Источником данных выбран DummyJSON

- Причина: API отдаёт готовый каталог товаров, категории, рейтинг, изображения и поддерживает пагинацию без авторизации.
- Следствие: можно быстро получить реалистичный каталог и сосредоточиться на MVVM/Flow логике.

### D-008. Категории сделаны динамическими, а не захардкоженными enum-значениями

- Причина: реальные категории API не укладываются в маленький фиксированный список.
- Следствие: `ProductCategory` хранит `slug` и `title`, а список chips строится по данным репозитория.

### D-009. Реализация репозитория пока остаётся внутри `feature`, без отдельного `data`-модуля

- Причина: для тестового это достаточно чисто и заметно проще по модульной сложности.
- Следствие: `model` хранит контракты, а `feature` временно содержит data-реализацию своей фичи.

### D-010. Для первой рабочей версии выбран локальный `load more` поверх заранее загруженного каталога

- Причина: это упрощает комбинирование поиска, категории и сортировки без каскада сетевых запросов на каждое действие пользователя.
- Следствие: приложение ходит в сеть за данными, но бизнес-логика фильтрации и пагинации остаётся предсказуемой и тестируемой.

## Открытые решения

### P-001. Нужен ли отдельный `data`-модуль

- Пока ответ: нет, для масштаба тестового это лишнее.
- Повод пересмотреть: если появится несколько источников данных или сильно вырастет data-layer.

### P-002. Нужен ли полноценный загрузчик изображений в shared UI

- Пока ответ: не внедрён в первом проходе.
- Повод пересмотреть: когда будем закрывать полировку UI под финальную сдачу.

## Принятые допущения

- Android остаётся основным демонстрационным клиентом для сдачи.
- KMP используется как архитектурное требование даже без отдельного iOS host-приложения.
- Если ТЗ оставляет свободу, выбирается решение, которое проще объяснить и тестировать.
