# Архитектурные решения

## Зачем этот файл

Здесь фиксируются инженерные решения по проекту, чтобы архитектура не расползалась,
а спорные места можно было нормально объяснить на собеседовании.

## Подтверждённые решения

### D-001. Проект будет переделан из Android-only шаблона в KMP-структуру

- Причина: это прямое требование задания
- Следствие: часть начальной работы будет про Gradle и модули, а не про экран

### D-002. `app` будет хост-модулем, а основная логика уйдёт в shared-модули

- Причина: так проще соблюсти KMP и не привязывать бизнес-логику к Android
- Следствие: ViewModel, state и логика фильтрации должны быть максимально общими

### D-003. Основное состояние экрана будет храниться в `StateFlow`

- Причина: экрану нужен воспроизводимый и наблюдаемый state с последним актуальным значением
- Следствие: UI будет подписываться на один источник правды

### D-004. Одноразовые события, если понадобятся, будут идти через `SharedFlow`

- Причина: transient-события не должны жить в постоянном UI state
- Следствие: ответ на вопрос про `StateFlow vs SharedFlow` будет опираться на реальный код

### D-005. Поиск, фильтрация, сортировка и комбинирование фильтров живут вне UI

- Причина: это основная бизнес-логика задания
- Следствие: её можно будет нормально тестировать без Android framework

### D-006. Тесты в первую очередь пишутся на shared-логику, а не на Android UI

- Причина: именно это проверяет задание
- Следствие: приоритет у `ViewModel` и use case тестов

### D-007. Все решения, принятые из-за неясностей ТЗ, должны быть задокументированы

- Причина: в задании прямо сказано, что требования можно уточнять инженерно
- Следствие: любое отклонение от шаблона нужно не скрывать, а объяснять

### D-008. На этапе init проект поднимается с Android как единственным host-клиентом

- Причина: нужно быстро собрать рабочий каркас под дальнейшую реализацию
- Следствие: shared-модули уже KMP, но демонстрация пока идёт через Android-приложение

### D-009. Shared UI стартует с placeholder-экраном для проверки связки модулей

- Причина: инициализационный коммит должен доказывать, что `app` уже подключён к `feature`
- Следствие: дальнейшая разработка пойдёт поверх готовой связки, а не с нуля

### D-010. Для shared-модулей используется Android-KMP library plugin

- Причина: это совместимый вариант для KMP-библиотек под текущий Android/Gradle стек
- Следствие: `model` и `feature` остаются root-модулями Gradle, а не папками внутри `app`

## Открытые решения

### P-001. Источник данных

- Варианты:
- внешний API
- локальный fake/stub набор данных
- смешанный вариант для разработки и демо
- Критерии:
- наличие категорий
- наличие картинок
- удобство пагинации
- предсказуемость схемы

### P-002. Модель пагинации

- Варианты:
- серверная пагинация
- локальная пагинация по заранее загруженному списку
- Критерии:
- простота
- объяснимость
- тестируемость

### P-003. Границы модулей

- Вопрос:
- ограничиться `model`, `feature`, `app`
- или добавить отдельный data-модуль
- Критерий:
- сохранить адекватную архитектуру без лишней сложности для тестового

### P-004. Ошибки и загрузочные состояния

- Вопрос:
- делать только happy path по минимуму
- или добавить полноценные loading/error/empty состояния
- Критерий:
- баланс между скоростью реализации и качеством решения

## Принятые допущения

- Android является основным целевым клиентом для сдачи
- KMP нужен как архитектурное требование, даже если демонстрируется Android-приложение
- Допустимо принять прагматичное решение, если оно зафиксировано и объяснимо
