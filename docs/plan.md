# План реализации Product Catalog

## Цель

Преобразовать текущий Android Compose-шаблон в аккуратный проект под тестовое
задание: с KMP-структурой, MVVM, тестируемой бизнес-логикой и нормальной сдачей.

## Общая стратегия

- Не генерировать весь проект одним большим куском
- Делать короткие завершённые этапы
- После каждого этапа иметь проверяемый результат
- Основную логику держать в shared-слое, а Android оставить host-слоем

## Этапы

### Этап 1. Перестройка структуры проекта

- Подготовить KMP-модули `model` и `feature`
- Оставить `app` как Android host-модуль
- Подключить нужные плагины и зависимости

### Этап 2. Базовая инфраструктура

- Подключить `Koin`
- Подключить `Ktor Client`
- Подключить `kotlinx.serialization`
- Подключить тестовые зависимости для корутин и моков

### Этап 3. Domain-слой

- Описать `Product`
- Описать `ProductCategory`
- Описать `SortOption`
- Описать `ProductRepository`

### Этап 4. Data-слой

- Выбрать источник данных
- Описать DTO
- Сделать маппинг DTO -> domain
- Реализовать репозиторий
- Зафиксировать стратегию пагинации

### Этап 5. Бизнес-логика

- Реализовать поиск
- Реализовать debounce `300ms`
- Реализовать фильтрацию по категории
- Реализовать сортировку
- Собрать всё через `Flow`

### Этап 6. Presentation-слой

- Сделать `ProductCatalogState`
- Сделать `ProductCatalogViewModel`
- Подключить DI
- Обеспечить единый источник правды для состояния экрана

### Этап 7. UI

- Сделать `ProductCatalogScreen`
- Сделать `SearchBar`
- Сделать `ProductCard`
- Сделать chips категорий
- Сделать пустое состояние
- Добавить `LazyColumn` и `load more`

### Этап 8. Тесты

- Написать тесты на `SearchProductsUseCase`
- Написать тесты на `ProductCatalogViewModel`
- Проверить init, поиск, фильтр, сортировку и комбинированные сценарии
- Отдельно проверить debounce и пагинацию

### Этап 9. Подготовка к сдаче

- Настроить `Detekt`
- Проверить `internal` visibility
- Привести форматирование к требованиям
- Подготовить README
- Подготовить скриншоты
- Записать видео
- Собрать APK и подготовить релиз

## Рекомендуемый порядок работы

1. Сначала привести проект к KMP-структуре.
2. Потом зафиксировать domain-модели и контракты.
3. Затем реализовать источник данных и репозиторий.
4. После этого собрать бизнес-логику и ViewModel.
5. Потом сделать UI.
6. Затем закрыть тесты.
7. В конце подготовить артефакты сдачи.

## Основные риски

- Перестройка в KMP может занять больше времени, чем сам экран
- Непродуманный источник данных может сломать пагинацию или категории
- Если сначала делать UI, а не shared-логику, тесты станут сложнее
- Если не фиксировать решения по ходу, потом будет трудно объяснять проект на собеседовании

## Правила работы

- Каждая задача должна заканчиваться проверкой
- Каждый спорный выбор нужно фиксировать в `docs/decisions.md`
- Не переходить к полировке UI, пока не собрана и не проверена основная логика
